13  Agentic AI and Next Steps

Debugging, critique, revision, reproducibility, and continued learning

Large language models can help with coding, data analysis, visualization, writing, and debugging. They can also produce confident mistakes. The goal is not to replace your judgment. The goal is to use AI as an assistant while keeping you responsible for the analysis.

This chapter focuses on practical workflows: how to ask for help, how to review answers, how to protect privacy, and how to use agentic tools without losing control of your project.

13.1 What AI can help with

AI tools are most useful when the task has enough structure for you to check the answer.

Common visualization uses include:

  • explaining unfamiliar R code
  • finding the source of an error message
  • suggesting a cleaner ggplot2 pipeline
  • proposing labels, captions, or alternative chart types
  • converting a rough plot into a more polished plot
  • reviewing a table or figure for readability
  • writing short helper functions
  • drafting Quarto prose around code output
  • checking whether a project depends on files that are not included

AI is less reliable when the task requires facts that may have changed, private data it cannot see, or judgment without enough context.

13.2 Web chatbots

Web chatbots are the most familiar AI interface. You paste a question, code, data description, or error message into a browser and get a response.

This works well for short, self-contained questions.

Example prompt:

I am using R and ggplot2. This code produces a plot, but the legend title is unclear.
Please explain what each line does and suggest a more readable version.

<paste code here>

For debugging, include the error message and the smallest code example that reproduces the problem.

I get this error in R:

<paste exact error>

Here is the code that triggered it:

<paste code>

Please identify the likely cause. Do not rewrite the whole script unless needed.

The main limitation of web chat is context. If the model cannot see your full project, it may guess incorrectly about data names, packages, or earlier steps.

13.3 IDE integration

Some AI tools work inside the coding environment. GitHub Copilot is one common example. In RStudio, Copilot can suggest completions while you write code. It can be useful for repetitive syntax, common transformations, and quick examples.

IDE integration has two advantages:

  • It can use nearby code as context.
  • It reduces copying and pasting between the editor and a browser.

It also has risks:

  • Suggested code may look plausible but use the wrong variable.
  • It may silently introduce a package you did not plan to use.
  • It may choose a more complicated approach than needed.

Treat inline suggestions as drafts. Read them before accepting them, and run the code in small pieces.

13.4 RStudio-focused AI with gander

The gander package is an RStudio-oriented way to send selected code and surrounding context to an LLM. It can be configured to use different model providers through packages such as ellmer.

The exact setup depends on the model provider and your local environment, so the following examples are not executed.

options(.gander_chat = ellmer::chat_openai(model = "gpt-4.1-mini"))
options(.gander_chat = ellmer::chat_google_gemini(model = "gemini-2.5-flash"))

The important workflow is:

  1. Select the code you want help with.
  2. Ask a focused question.
  3. Review the response before applying it.
  4. Run the revised code.
  5. Compare the output to what you expected.

Useful requests include:

Explain this pipeline one step at a time.
Suggest a clearer version of this plot, but keep the same data and variables.
Find likely bugs in this code. Do not change the statistical model.

13.5 Programmatic AI with ellmer

ellmer gives R users a more programmatic interface to language models. Instead of using a web page manually, you can make model calls from R code.

This can be useful for research workflows such as:

  • summarizing text fields
  • classifying open-ended survey responses
  • generating draft labels for qualitative coding
  • extracting structured information from documents
  • comparing the behavior of multiple prompts

Programmatic AI needs extra care. Save prompts, model names, settings, dates, and outputs when they are part of the research process. If an LLM contributes to an analysis, the workflow should be documented like any other computational step.

library(ellmer)

chat <- chat_openai(model = "gpt-4.1-mini")

chat$chat(
  "In one sentence, explain why direct labels can be easier to read than legends."
)

Do not send confidential, identifiable, or restricted data to a cloud model unless your data agreement and institutional rules allow it.

13.6 Local and cloud models

Cloud models run on servers provided by companies or institutions. Local models run on your own computer or a controlled server.

Cloud models often provide stronger performance and easier setup. They can be accessed through web interfaces, IDE tools, or APIs.

Local models can improve privacy and reproducibility because your data do not need to leave your machine and you can pin a specific model version. Tools such as Ollama make local models easier to run, but performance depends on hardware. Larger models may require a powerful GPU or a recent machine with enough memory.

The practical choice depends on the project:

  • Use cloud models for general learning, public examples, and low-sensitivity code help.
  • Consider local models for sensitive data, restricted research material, or workflows where you need control over the exact model.
  • Avoid sending private data, unpublished research data, student records, or confidential text to a public chatbot.

13.7 Agentic coding tools

Agentic tools go beyond chat. They can inspect files, run commands, edit code, run tests, and iterate. Examples include terminal or codebase agents such as Claude Code, Codex, Gemini CLI, and similar tools.

The difference from a chatbot is scope. A chatbot usually answers from pasted context. An agentic tool can work inside a project.

This makes agents useful for:

  • finding where a function or variable is defined
  • updating several related files
  • running checks after edits
  • comparing source and rendered output
  • writing tests or validation scripts
  • reviewing a project for missing files or missing packages

The same power creates risk. An agent can make broad changes quickly. Use boundaries.

Good instructions for an agent are explicit:

Edit only 07-lines-bars-and-annotation.qmd.
Do not modify data files or setup files.
Use files from the course project, especially files in Data/.
After editing, report exactly which files changed.

For code review:

Review this Quarto chapter for runnable R code, missing files, missing packages, and unclear explanations.
Do not edit files. List findings by severity with file and line references.

For debugging:

The render fails in 12-mapping.qmd.
Find the smallest change that fixes the render.
Do not change other chapters.
Explain the cause and the verification command.

13.8 A safe AI workflow

Use AI in small loops.

  1. State the task and constraints.
  2. Provide only the context needed.
  3. Ask for a small change or diagnosis.
  4. Inspect the answer before applying it.
  5. Run the code.
  6. Compare the output to the goal.
  7. Commit or save only after verification.

This loop is slower than accepting the first suggestion, but it prevents many common errors.

13.9 How to ask better questions

A strong AI request includes the goal, context, constraints, and desired output.

Weak request:

Make this plot better.

Stronger request:

This plot compares life expectancy by continent in 2007 using Gapminder.
Suggest improvements that make the comparison clearer for a general audience.
Keep the data and chart type the same.
Return revised ggplot2 code and explain the changes briefly.

Weak request:

Fix my code.

Stronger request:

This Quarto chunk should read Data/gapminder/gapminder.csv and make a scatterplot.
It fails with the error below.
Identify the likely cause and propose the smallest code change.
Do not introduce new packages.

<paste error>
<paste code>

13.10 Reviewing AI output

Before using AI-generated code, check:

  • Does it use variables that exist?
  • Does it use packages that are loaded or installed?
  • Does it preserve the original research question?
  • Does it change the model, filter, or grouping without saying so?
  • Does it depend on files that are not included with the project?
  • Does it silently drop missing data?
  • Does the plot or table support the intended interpretation?
  • Can you explain every line?

If you cannot explain the code, ask the model to explain it or simplify it before using it.

13.11 Privacy, attribution, and reproducibility

AI use is part of the research workflow. Treat it like other research assistance.

Protect privacy:

  • Do not paste sensitive data into public tools.
  • Remove names, IDs, free-text responses, and confidential details unless sharing is allowed.
  • Check institutional, funder, journal, and IRB rules.

Support reproducibility:

  • Save important prompts when they affect analysis decisions.
  • Record model names when AI output is part of a method.
  • Keep AI-generated code in scripts or Quarto documents where it can be reviewed.
  • Prefer deterministic code over one-off manual AI interactions for repeated tasks.

Use attribution when appropriate. If AI contributed materially to text, code, or analysis, follow the disclosure rules for the course, journal, employer, or institution.

13.12 AI for visualization critique

AI can be useful as a first-pass critic of a figure. Ask it to evaluate the plot against specific criteria rather than asking whether it is “good.”

Review this visualization for a social science audience.
Focus on:
1. whether the chart type matches the comparison,
2. whether labels and legends are clear,
3. whether the color scale is appropriate,
4. whether the caption supports the main claim.

Do not invent facts beyond what is visible in the plot.

When possible, provide the code and the rendered figure. The code shows how the chart was made. The image shows what the reader sees.

13.13 AI for debugging Quarto projects

Quarto errors often come from one of a few causes:

  • a missing package
  • a data file that does not exist
  • a chunk that depends on an object created in a different chapter
  • a typo in a variable name
  • output written to a missing folder
  • internet-dependent code running during render
  • YAML indentation errors

A useful AI debugging prompt is specific:

This Quarto render fails.
Here is the error message and the chunk before it.
Please identify the most likely cause and propose one minimal fix.
If the problem is a missing data file, suggest how to check it in R.

Then verify the answer in R:

file.exists("Data/cigarettes/cigarettes.csv")
renv::status()
quarto::quarto_render("12-mapping.qmd")

13.14 Next steps after the course

The course has covered a full workflow:

  • import data
  • tidy and transform data
  • build plots with ggplot2
  • separate and compare groups
  • choose chart types
  • annotate and highlight
  • create tables
  • visualize model output
  • make maps
  • use interactivity and Quarto output formats
  • use AI assistance critically

The next step is to apply the workflow to one project you care about.

Choose a dataset and make a small portfolio:

  1. One exploratory table.
  2. One polished static plot.
  3. One model visualization or map, if appropriate.
  4. One short Quarto report explaining the question, data, code, and result.

Keep the project portable. Keep data inside the project, put code in scripts or Quarto files, and render the report from a clean session.

13.15 Short exercise

Choose one plot or table from an earlier chapter.

  1. Write a prompt asking an AI tool to critique it.
  2. Include at least three constraints in the prompt.
  3. Ask for suggestions, not a full rewrite.
  4. Decide which suggestions you would accept, reject, or test.
  5. Write two sentences explaining your decision.

13.16 Final checklist

Before sharing a visualization project, check:

  • The project renders from a clean session.
  • Data files are included with the project.
  • Required packages are listed.
  • Data sources are documented.
  • Tables and figures have clear titles, labels, and captions.
  • Optional internet/API steps are marked clearly.
  • AI-assisted code has been reviewed and tested.
  • The final report explains both what you found and how you produced it.

The purpose of the workflow is not just to make a figure. It is to make a figure that another person can understand, evaluate, and reproduce.
